## 0. 요약 (Executive Summary)

현재 시스템은 **빠르게 동작하는 PoC** 수준이다. CAT 로직, AI 대화, 이메일 리포트의 핵심 플로우는 완성되어 있고 Railway에 배포·운영 중이다. 그러나 **JSON 파일 기반 단일 서버 구조**라는 근본적인 제약이 있어, 동시 사용자 증가 / 학원 확대 / 기능 추가 시 반드시 리팩터링이 필요하다. 또한 실제 잉글랩 지필 문항(10종 299문항)이 `data/source-materials/englab_item_bank.json`에 완전 구조화되어 있지만, 현재 앱의 `public/questions.json`과는 **완전히 분리**되어 있어 연결이 안 된 상태다.

## 4. 기능별 진단

### 4-1. CAT (Computer Adaptive Test) 로직 — `app.js:8~135`

**구조:** `CATEngine` 클래스. 4개 섹션을 순차적으로 진행(Vocabulary 10 → Grammar 10 → Reading 5 → Writing 5 = 총 30문항). 섹션 내에서만 CAT 적용.

**theta 업데이트 방식:**
```
lrSec = max(0.2, 0.6 − n × 0.04)   // 문항 수 증가에 따라 학습률 감소
tSec  = difficulty ± 0.8             // 정답이면 +0.8, 오답이면 -0.8
sectionTheta += lrSec × (tSec − sectionTheta)
globalTheta  = vocab×0.35 + grammar×0.35 + reading×0.20 + writing×0.10
```

**진단:**

| 항목 | 평가 | 내용 |
|------|------|------|
| 섹션 독립 theta | ✅ 적절 | writing 오답이 전체 theta를 끌어내리지 않도록 가중치 10% 제한 |
| 난이도 선택 범위 | ⚠️ 주의 | `delta 0~5` 확장 탐색 — 문항풀이 얕으면 delta 5까지 가서 부적합한 난이도 출제 가능 |
| 조기 종료 없음 | ⚠️ 고려 | theta 확신도 높아도 무조건 30문항 고정 — Inventor급 학생도 30문항 전부 응시 |
| 섹션 순서 고정 | ✅ 의도적 | 항상 vocab→grammar→reading→writing 순서 |
| Seeker 초급 처리 | ✅ 적절 | sectionTheta ≤ 1.5이면 처음 4문항을 그림-단어 우선 배정 |
| `Math.random()` 반복 | ⚠️ 주의 | `sort(() => Math.random() - 0.5)` 사용 — 공정하지 않은 셔플 (Knuth 셔플 권장) |

**실제 IRT vs 현재 구현 차이:**  
실제 IRT(Item Response Theory)는 Fisher Information 최대화로 문항을 선택하고, 능력 추정에 MLE/EAP를 사용한다. 현재 구현은 "난이도 타겟팅 + 학습률 감소" 방식의 단순화 모델이다. PoC에는 충분하나, 문항 노출/편향 분석이 없어 운영 데이터 누적 후 검증 필요.

### 4-2. 문항 배정 로직 — `public/questions.json`

**현황:** 수동으로 제작한 문항 뱅크. 정확한 총 문항 수는 JSON 파일 직접 집계 필요하나, CAT 운영에 필요한 최소 풀은 섹션별 × 난이도별 최소 2~3문항.

**문제점:**

1. **잉글랩 실제 문항과 단절**: `englab_item_bank.json`의 299개 지필 문항이 현재 앱에 전혀 사용되지 않는다. 현재 `questions.json`은 별도로 제작된 것으로 보인다.

2. **난이도 분포 불명**: questions.json의 섹션별 × 난이도별 분포가 검증되지 않았다. CAT에서 delta가 커지면(±4, ±5) 타겟 난이도와 동떨어진 문항이 출제될 수 있다.

3. **Reading 지문 구조**: item_bank의 `passage_id` 참조 방식과 현재 questions.json의 구조가 다를 수 있다 — 지문 공유 여부 확인 필요.

4. **Writing 평가 신뢰도**: `submitWriting()`이 문자열 정규화(소문자, 부호 제거) 후 `expected[]` 배열과 비교. 완전 일치 방식이라 동의어/복수 정답 처리에 한계.

### 4-3. AI 대화 평가 — `server.js:126~311`

**흐름:**
1. 클라이언트가 매 턴마다 `POST /api/chat` 호출 (메시지 히스토리 + mcqTheta 전달)
2. 서버가 theta 기반 동적 시스템 프롬프트 생성
3. Claude Sonnet 4.6에 전달 (비-최종 턴: max_tokens=80, 최종 턴: 600)
4. 최종 턴(turn 5)에서 JSON 평가 추출 → `assessment_complete:true` 마커로 파싱

**레벨별 프롬프트 전략:**

| theta 범위 | 레벨 | 전략 |
|-----------|------|------|
| < 2 | Seeker | 이모지 퀴즈(A) → either/or(B) → 자유 질문(C) — 3단계 점진 |
| 2~4 | Builder | 짧은 일상 영어 질문, 한국어 칭찬 |
| 4~6 | Challenger | 영어 전용, 2문장 제한 |
| 6~8 | Explorer | 영어 전용, 간결한 반응 |
| ≥ 8 | Inventor | 분석적 질문, 2문장 |

**진단:**

| 항목 | 평가 | 내용 |
|------|------|------|
| 레벨 상대 평가 | ✅ 적절 | "자기 레벨 대비" 평가 철학 명시 (Seeker 완벽 응답 → 8-9점) |
| few-shot 예시 | ✅ 효과적 | 첫 턴 포맷 강제에 few-shot 사용 |
| JSON 파싱 | ⚠️ 취약 | 정규식 기반 추출, Claude가 마크다운 블록에 감쌀 경우 실패 가능 (현재 strip 처리 있음) |
| max_tokens=80 | ⚠️ 위험 | 비-최종 턴 응답을 80토큰으로 제한 — Claude가 말 중간에 잘릴 수 있음 |
| 대화 로그 저장 | ✅ 완비 | 전체 conversation을 results.json에 저장 |
| final_level 무시 | ✅ 의도적 | Claude가 final_level 필드 출력하지만 실제 level 결정은 mcqTheta만 사용 (일관성 유지) |
| 에러 처리 | ⚠️ 부족 | Claude API 500 시 "오류 메시지" 반환 후 플로우 중단 — 재시도 로직 없음 |

## 5. 소스 문항 뱅크 분석 (`englab_item_bank.json`)

### 5-1. 현황 요약

| 구분 | 시험지 수 | 문항 수 | 상태 |
|------|-----------|---------|------|
| 완전 구조화 | 8종 | ~239개 | Challenger/Explorer/Inventor/Innovator (1-6 / 7-12) — 즉시 사용 가능 |
| 검수 필요 | 2종 | ~60개 | Seeker(29/30 파싱), Builder — 이미지 의존, 수기 검수 필요 |
| **합계** | **10종** | **299개** | |

### 5-2. 섹션 구성 (Seeker 제외 8종 기준)

각 시험지는 **30문항, 섹션 구성 동일**: vocabulary + grammar + reading(지문 포함) + writing(unscramble)

### 5-3. 이미지 파일 현황

`data/source-materials/images/`에 Seeker + Builder 이미지가 있어야 하나, 현재 디렉토리에 파일이 0개로 확인됨. 이미지 파일이 누락되었거나 아직 추가되지 않은 상태.

### 5-4. item_bank → questions.json 변환 시 필요한 작업

| 항목 | item_bank | questions.json | 변환 작업 |
|------|-----------|----------------|-----------|
| 정답 형식 | `"a"/"b"/"c"` 문자열 | `0/1/2/3` 인덱스 | `{a:0, b:1, c:2, d:3}[answer]` |
| 섹션명 | `"vocabulary"/"grammar"/"reading"/"writing"` | 동일 | 호환 |
| 보기 형식 | `[{key:"a", text:"..."}]` | `["...","..."]` | `options.map(o => o.text)` |
| 지문 연결 | `passage_id` 참조 | 미지원 | Reading 지문을 question.passage 필드로 내재화 |
| unscramble | `chunks[]` + `answer` 문장 | `words[]` + `expected[]` | chunks → words, answer → expected[0] |
| difficulty | 없음 (레벨 × 번호 추론) | 0~10 정수 | 레벨 매핑 테이블로 할당 필요 |
| image_dependent | `needs_review:true` | questionType="picture" | 이미지 경로 연결 + 수기 검수 후 별도 처리 |

**가장 중요한 누락: difficulty 필드**. item_bank에 문항별 difficulty가 없다. CAT는 difficulty로 선택하므로, 시험지 레벨 + 번호 순서로 difficulty를 추론·할당하는 로직이 필요하다.

```
Seeker  문항 → difficulty 0~2
Builder 문항 → difficulty 2~4
Challenger 문항 → difficulty 4~6
Explorer 문항 → difficulty 6~8
Inventor 문항 → difficulty 8~10
Innovator 문항 → difficulty 9~10
```

## 6. Top 3 리스크

### 리스크 1 (심각도: 높음) — JSON 파일 Race Condition

**문제:** 동시 완료 요청 시 결과 유실 가능.

```
시각 T:  서버가 results.json 읽음 (10개 항목)
시각 T+1ms: 다른 요청도 results.json 읽음 (동일하게 10개 항목)
시각 T+5ms: 첫 요청이 11번째 항목 추가 후 저장
시각 T+6ms: 두 번째 요청이 11번째 항목 추가 후 저장 → 첫 요청 결과 덮어쓰기
```

단일 학원 소규모 운영 중에는 발생 빈도 낮으나, 여러 학원이 동시에 테스트를 실시하면 데이터 유실이 발생한다.

**단기 해결:** `writeJSON` 시 파일 잠금(예: `proper-lockfile` 패키지) 또는 append-only 방식.  
**근본 해결:** SQLite 또는 PostgreSQL 도입 (Railway PostgreSQL 서비스 추가).

---

### 리스크 2 (심각도: 높음) — 실제 잉글랩 문항 미연결

**문제:** `englab_item_bank.json`(10종 299문항, 학원 실제 사용 문항)이 앱에 전혀 연결되지 않았다. 현재 앱이 사용하는 `questions.json`은 별도 제작된 것으로, 학원의 실제 교육 내용과 다를 수 있다. 이로 인해:
- 레벨 배치 결과와 실제 교재 배치가 불일치할 가능성
- item_bank의 `diagnostic_table` (점수→교재 추천)을 활용하지 못함
- 8종 완전 구조화 문항 + 진단표가 사장되고 있음

**해결 방향:** 아래 7절 데이터 설계안 참조.

---

### 리스크 3 (심각도: 중간) — 관리자 인증 보안

**문제:**
1. 비밀번호가 URL 파라미터(`?pw=xxx`)로 전달됨 → 서버 액세스 로그, 브라우저 히스토리, Referer 헤더에 노출
2. 기본값 `englab2024`가 코드에 하드코딩 (server.js:677)
3. Rate limiting 없음 → 비밀번호 브루트포스 무방비

**단기 해결:**
- 쿼리 파라미터 대신 `Authorization: Bearer <pw>` 헤더 사용
- 기본 비밀번호 제거, 환경변수 필수화
- 5회 실패 시 IP 잠금 (express-rate-limit)

## 7. 데이터 설계안 (다음 단계 제안)

### 7-1. 문항 뱅크 통합 방안

`englab_item_bank.json` → `public/questions.json` 변환 파이프라인:

```
data/source-materials/englab_item_bank.json
          ↓  (Node.js 변환 스크립트, 1회성 실행)
public/questions.json  (앱이 로드하는 CAT 문항 풀)
```

**변환 스크립트 (`scripts/build-questions.js`)가 해야 할 일:**

1. `image_dependent:false` 인 8종 문항만 처리 (Seeker/Builder는 이미지 검수 후 별도)
2. answer 변환: `"a"→0, "b"→1, "c"→2, "d"→3`
3. options 변환: `[{key,text}]` → `[text,text,...]`
4. difficulty 할당: 레벨별 매핑 테이블 + 문항 번호 내 순차 미세 조정

   ```javascript
   const LEVEL_DIFFICULTY = {
     Challenger: [4, 5, 6],   // 난이도 범위
     Explorer:   [6, 7, 8],
     Inventor:   [7, 8, 9],
     Innovator:  [8, 9, 10],
   };
   ```

5. Reading 지문 내재화: `passage_id` → `question.passage` 필드에 지문 텍스트 포함
6. Writing unscramble: `chunks[]` → `words[]`, `answer` → `expected[0]`
7. questionType 할당: `multiple_choice→"multiple_choice"`, `unscramble→"unscramble"`, `image_dependent→"picture"`(이미지 연결 후)

**예상 산출물:**
- 완전 구조화 8종 × 30문항 = **240문항** 즉시 활용 가능
- 현재 `questions.json`과 병합하면 더 풍부한 풀 구성 가능

### 7-2. diagnostic_table 활용 방안

각 시험지의 `diagnostic_table`을 활용해 리포트에 "추천 교재" 정보를 추가할 수 있다:

```javascript
// 현재: finalLevel = thetaToLevelString(mcqTheta)
// 추가: 해당 레벨 시험지의 diagnostic_table에서 권장 교재 룩업
const diagnosis = findDiagnosis(level, score, diagnosticTable);
// → { recommended: "Challenger 7-12", diagnosis: "우수" }
```

이 정보를 이메일 리포트와 어드민 화면에 추가하면 학원 선생님에게 직접적인 교재 배치 근거를 제공할 수 있다.

### 7-3. 중장기 DB 전환 설계 (참고)

운영 학원 3개 이상 / 일 100명 이상 시점에 전환 권장:

```
PostgreSQL (Railway)
├── students (id, name, grade, academy_id, created_at)
├── academies (id, name, contact_email, added_at)
├── test_sessions (id, student_id, session_key, started_at, completed_at, duration_sec)
├── mcq_results (session_id, theta, correct, total, by_section jsonb)
├── ai_assessments (session_id, vocab, grammar, complexity, overall, strengths, growth, ...)
├── chat_messages (session_id, role, content, turn_no, created_at)
├── event_logs (id, session_id, event_type, ip, ua, extra jsonb, created_at)
└── questions (id, section, type, difficulty, stem, options jsonb, answer, ...)
```

JSON → PostgreSQL 마이그레이션은 현재 `results.json`을 그대로 시드 데이터로 사용 가능 (필드 대응 1:1).
